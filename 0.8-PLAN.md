# 0.8 RAG Reliability And Efficiency Plan

Goal: make RAG indexing and retrieval more relevant, efficient, extensible, and resilient for production Laravel applications.

Status: implementation and local hardening complete; live pgvector/Qdrant integration validation and release tagging pending.

## Design Direction

The RAG orchestration, contracts, configuration, and Laravel integrations should live in this package. Content-specific behavior should remain extensible through strategies and adapters so applications can register strict strategies for their frequently handled documents without coupling the core to one document set.

This creates two complementary paths:

- Graceful defaults for regular text, code, and common document types.
- Application-defined strategies for domain- or team-specific documents where structure is known in advance.

As the strategy catalog grows, common indexing workflows should become faster, less expensive, and more accurate.

## 1. Content-Aware Chunking

- Add a chunking strategy router that selects a strategy using MIME type, file extension, metadata, or an explicit override.
- Keep recursive character chunking as the general fallback.
- Add semantic natural-language splitting that preserves headings, paragraphs, lists, and sentence boundaries.
- Add syntax-aware code splitting that preserves functions, classes, methods, and other meaningful declarations where practical.
- Allow applications to register custom strategies for known document families.
- Record the selected strategy and strategy version in chunk metadata for reproducible re-indexing.

## 2. Incremental Indexing And Embedding Cache

- Add an embedding cache contract with Laravel cache and persistent-store adapters.
- Key cached embeddings by content hash, embedding provider, model, dimensions, and normalization version.
- Skip embedding API calls for unchanged chunks while still reconciling vector-store records.
- Expose cache hit, miss, saved-call, and saved-token metrics in indexing summaries and observability spans.
- Define invalidation behavior when the model, dimensions, chunking strategy, or normalization rules change.

## 3. Retrieval Relevance And Empty Results

- Add configurable global, per-retriever, and per-query minimum similarity scores.
- Filter semantically distant results before they are exposed to agents or workflows.
- Return an explicit empty result when no result meets the threshold.
- Let agent tools define a zero-result policy, such as asking for clarification, using another approved source, or answering that relevant context was not found.
- Ensure retrieval prompts do not imply that context exists when the result set is empty.
- Keep thresholds store-independent while documenting that score calibration may vary by embedding model and vector store.

## 4. Queue-Based And Resumable Indexing

- Add Laravel queue jobs for document loading, chunking, embedding, and vector-store writes.
- Support retries, backoff, timeouts, failure reporting, and idempotent reprocessing.
- Split large indexing work into bounded batches to avoid HTTP request timeouts and excessive worker memory usage.
- Preserve the synchronous indexer for small documents, tests, and command-line workflows.
- Add progress and completion events so applications can expose indexing status.
- Define safe replacement behavior so a failed re-index does not remove the previously available document.

## 5. Large-File And Streaming Behavior

- Keep the current explicit rejection of files above the configured byte limit; do not silently truncate them.
- Add optional streaming loaders and chunkers for supported formats so large documents can be processed without loading the complete file into memory.
- Preserve sentence, paragraph, or syntax boundaries across stream buffers.
- Define limits for source bytes, extracted text, chunk count, and per-chunk size.
- Report clear failures for unsupported binary files, malformed encodings, and partially readable sources.

## 6. Metadata Schema And Filter Safety

- Define and document the metadata value types supported consistently across vector stores.
- Normalize metadata keys and values before indexing and filtering.
- Add optional metadata schema validation for applications that require strict fields and types.
- Reject unsupported nested filters and ambiguous values with clear validation errors.
- Continue using parameterized pgvector queries and validate configurable identifiers.
- Add cross-store tests for special characters, Unicode, nested objects, null values, arrays, and invalid JSON-serializable values.

## 7. Proposed Public API

```php
LaravelAgents::indexAsync(
    loader: new StreamingFileDocumentLoader(storage_path('knowledge/large.md')),
    namespace: 'product-docs',
    queue: 'rag-indexing',
    batchDocuments: 25,
);
```

```php
$results = LaravelAgents::retriever(namespace: 'product-docs')->retrieve(
    query: 'How do agents call tools?',
    limit: 5,
    minimumScore: 0.75,
);
```

```php
'rag' => [
    'chunking' => [
        'default' => 'semantic',
        'strategies' => [
            'team-docs' => App\Rag\TeamDocumentChunkingStrategy::class,
        ],
    ],
    'retrieval' => [
        'minimum_score' => 0.75,
        'zero_result_policy' => 'explicit_empty',
    ],
],
```

The exact API should remain additive and may change during implementation before v1.0.

## 8. Tests And Documentation

- Unit tests for strategy selection, boundary preservation, cache keys, thresholds, and metadata validation.
- Integration tests for queued retries, resumable indexing, stale-record replacement, and zero-result agent behavior.
- Large-file tests that verify explicit rejection and streaming behavior without truncation.
- Compatibility tests across in-memory, pgvector, and Qdrant stores.
- Benchmarks for embedding cache savings, indexing throughput, memory use, and retrieval relevance.
- Complete English and pt-BR usage documentation with examples for custom document strategies.

## Open Questions

- Which document formats and programming languages should receive built-in strategies first?
- Which team-specific or domain-specific documents are frequent and structured enough to justify strict predefined strategies?
- Should queued indexing be enabled explicitly, selected automatically above a size threshold, or both?
- Which cache adapter should be the default, and how long should embeddings be retained?
- Should similarity thresholds have one package default or require calibration per embedding model and namespace?
- Which nested metadata operations, if any, must be portable across pgvector and external vector stores?
- What should the default zero-result behavior be for `RetrieverTool` while keeping agents honest about missing context?

## Implementation Order

1. Similarity thresholds, explicit empty results, and zero-result tool behavior.
2. Chunking strategy contract, router, and content-aware built-in strategies.
3. Embedding cache contract, Laravel adapter, metrics, and invalidation rules.
4. Metadata schema enforcement and cross-store filter compatibility.
5. Queue jobs, events, retries, progress reporting, and resumable indexing.
6. Streaming loaders and large-file processing.
7. Benchmarks, tests, and EN/pt-BR documentation.
8. Final validation, commit, and tag `0.8`.
